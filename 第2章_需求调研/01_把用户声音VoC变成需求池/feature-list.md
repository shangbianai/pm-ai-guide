## 产品定位

面向连锁餐饮品牌（初期 15 家门店），解决门店销售数据上报依赖微信群和 Excel 导致统计混乱、库存不透明频繁断货的核心问题，提供总部–门店双向的数字化运营管理平台，支持 PC + 微信小程序使用。

**设计原则（v1.1 补充）：** 系统需极简易用，以一线员工为设计基准，核心功能操作步数参照即时通讯应用标准；员工无需培训即可独立完成打卡、查排班等日常操作。

## 核心功能（P0）

### F-001：用户登录与身份认证
- **用户故事**：作为系统任意用户，我需要通过账号密码（或手机验证码）登录，以便系统识别我的身份并控制数据访问范围
- **Kano 类型**：基本型（系统基础设施，不参与优先级评分）
- **核心逻辑**：账号密码登录 + 手机号验证码登录；会话过期自动跳转登录页；密码加密存储；移动端支持登录状态保持
- **验收标准**：
  - [ ] 账号密码正确时，5 秒内完成登录跳转
  - [ ] 密码连续错误 5 次，账号锁定 15 分钟并给出明确提示
  - [ ] 移动端登录后，7 天内无需重复登录（记住登录状态）
- **依赖**：无

### F-002：角色权限体系
- **用户故事**：作为系统管理员，我需要为不同角色（总部管理员 / 店长 / 员工）设置权限，以便不同人员只能访问其职责范围内的数据
- **Kano 类型**：基本型（系统基础设施，不参与优先级评分）
- **核心逻辑**：三级角色：总部管理员（全店数据读写 + 用户管理）、店长（本店数据读写 + 员工管理）、员工（只读排班 / 打卡）；门店数据默认隔离，店长只能看本店
- **验收标准**：
  - [ ] 员工角色登录后，无法看到其他门店或总部汇总数据
  - [ ] 店长可管理本店员工，但无法修改其他门店数据
  - [ ] 总部管理员可查看所有门店数据，可创建 / 停用账号
- **依赖**：F-001

### F-003：门店日销售数据录入
- **用户故事**：作为门店店长，我需要每天录入本店的营业额、客单价、翻台率，以便总部能及时掌握门店经营状况，不再靠微信群汇报
- **Kano 类型**：基本型
- **核心逻辑**：每天每店填一条记录；字段：日期（默认今天）、营业额（元）、客单价（元）、翻台率（次/桌）；当天内可修改，次日 09:00 自动锁定；提交后推送总部通知
- **验收标准**：
  - [ ] 店长可在 3 步以内完成当日数据录入并提交
  - [ ] 录入非法格式（负数 / 非数字）时实时给出错误提示，不允许提交
  - [ ] 当日数据次日 09:00 前自动锁定，不可再修改
  - [ ] 数据成功提交后，总部看板 1 分钟内可见
- **依赖**：F-001、F-002

### F-004：总部销售数据汇总看板
- **用户故事**：作为总部管理人员，我需要在一个页面看到所有门店的今日 / 本周 / 本月销售数据，以便不再依赖微信群和 Excel 做统计
- **Kano 类型**：基本型
- **核心逻辑**：默认展示今日所有门店营业额排名；支持切换日 / 周 / 月视图；汇总合计 + 门店对比；未录入数据的门店标红提醒；支持下钻到单店明细
- **验收标准**：
  - [ ] 页面加载完成，所有已录入门店数据在 3 秒内展示完毕
  - [ ] 能一眼识别哪些门店当日尚未录入数据（颜色或角标标记）
  - [ ] 点击任意门店可查看该店的历史录入明细
- **依赖**：F-001、F-002、F-003

### F-007：门店库存日常记录
- **用户故事**：作为门店店长，我需要每天记录各食材的入库量、消耗量和当前余量，以便掌握库存状态，防止断货影响营业
- **Kano 类型**：基本型
- **核心逻辑**：维护门店食材品类列表；每日填写入库量和消耗量，系统自动计算余量（余量 = 上日余量 + 入库 - 消耗）；首次使用时可录入初始库存
- **验收标准**：
  - [ ] 系统根据入库和消耗自动计算余量，无需手动填写余量
  - [ ] 首次使用时，店长可录入初始库存数据（存量迁移）
  - [ ] 录入消耗量导致余量为负时，给出明显警告提示（但允许提交）
- **依赖**：F-001、F-002

### F-008：库存低位预警
- **用户故事**：作为门店店长，我需要在某食材库存低于安全库存线时收到提醒，以便提前补货，避免断货影响营业
- **Kano 类型**：期望型（客户明确提出且已发生多次断货，实际重要性等同于基本型）
- **核心逻辑**：每个食材可设置安全库存阈值；余量 ≤ 阈值时通过系统消息 / 推送通知店长；总部管理员可查看全网预警汇总；余量恢复后预警自动解除
- **验收标准**：
  - [ ] 触发预警后，店长在 5 分钟内收到系统通知（App 推送或消息中心）
  - [ ] 总部管理员可在一个页面看到全网所有低库存预警项
  - [ ] 库存补货后余量超过阈值，预警自动解除，不再显示为异常
- **依赖**：F-001、F-002、F-007

## 重要功能（P1）

### F-005：销售趋势分析
- **用户故事**：作为总部管理人员，我需要查看各门店按周 / 月 / 季度的销售趋势图，以便分析经营走势并做决策
- **Kano 类型**：期望型
- **核心逻辑**：折线图展示选定门店或全品牌的营业额 / 客单价趋势；支持按周、月、季度切换；支持多店对比（最多 5 店）
- **验收标准**：
  - [ ] 可选择单店或多店（最多 5 店）进行趋势对比
  - [ ] 趋势图加载时间 ≤ 3 秒
  - [ ] 数据有缺失日期时，图表中明显标注（断点或灰色区间）
- **依赖**：F-003、F-004

### F-006：销售数据导出 Excel
- **用户故事**：作为总部管理人员，我需要将选定时间段内所有门店的销售数据导出为 Excel，以便离线分析或向管理层汇报
- **Kano 类型**：期望型
- **核心逻辑**：选择门店（全部或指定）和时间范围后导出；字段：日期、门店名称、营业额、客单价、翻台率；格式为 .xlsx
- **验收标准**：
  - [ ] 点击导出按钮，文件在 10 秒内生成并自动下载
  - [ ] 导出数据与看板展示数据一致，零误差
  - [ ] 文件名格式为"销售数据_开始日期_结束日期.xlsx"
- **依赖**：F-003

### F-009：总部库存总览
- **用户故事**：作为总部管理人员，我需要查看各门店当前库存状态，以便统筹安排补货计划和物资调配
- **Kano 类型**：期望型
- **核心逻辑**：按门店展示关键食材当前余量和预警状态；支持筛选"有预警"门店；总部不可直接修改门店库存数据
- **验收标准**：
  - [ ] 总部可看到所有门店的库存汇总，含余量和是否触发预警
  - [ ] 点击任意门店可查看该店库存明细
  - [ ] 支持按"预警状态"筛选，快速定位问题门店
- **依赖**：F-007、F-008

### F-010：员工档案管理
- **用户故事**：作为门店店长，我需要在系统中录入和维护本店员工的基本信息，以便排班和考勤统计有准确的人员数据
- **Kano 类型**：基本型（F-011、F-012 的前置依赖，不独立产生业务价值）
- **核心逻辑**：店长增删改查本店员工档案；字段：姓名、手机号、岗位、入职日期、在职状态；总部可查看所有门店员工档案；离职员工标记而非删除
- **验收标准**：
  - [ ] 店长可在 1 分钟内完成一名新员工信息的录入
  - [ ] 同一门店内手机号唯一，重复录入时给出提示
  - [ ] 离职员工可标记为"已离职"，历史记录不删除
- **依赖**：F-001、F-002

### F-011：排班管理
- **用户故事**：作为门店店长，我需要在系统中制定员工排班并发布，以便员工能随时用手机查看自己的班次，减少微信通知的混乱
- **Kano 类型**：期望型
- **核心逻辑**：店长按周创建排班表；为每个班次指定员工；发布后员工可在手机端查看；排班变更后被调整员工收到通知
- **验收标准**：
  - [ ] 店长完成一周排班并发布，操作步骤 ≤ 5 步
  - [ ] 员工打开 App 后，首页默认展示"我的下一班次"
  - [ ] 排班发生变更时，被调整的员工收到 App 推送通知
- **依赖**：F-001、F-002、F-010

### F-012：员工打卡
- **用户故事**：作为普通员工，我需要通过手机 App 完成上下班打卡，以便考勤记录自动生成，无需店长手动统计
- **Kano 类型**：期望型
- **核心逻辑**：员工在指定时间窗口内打卡（上班 / 下班各一次）；打卡方式待确认（见 Q-001）；系统自动记录打卡状态（正常 / 迟到 / 早退 / 缺卡）
- **验收标准**：
  - [ ] 员工打卡操作在 2 步以内完成（开 App → 确认打卡）
  - [ ] 打卡成功后，考勤记录在 1 分钟内可在店长端查看
  - [ ] 员工可查看本人当月打卡明细，响应时间 ≤ 2 秒
- **依赖**：F-001、F-002、F-010
- **注**：打卡方式（GPS / 二维码 / 两者）影响技术复杂度，需确认，见 Q-001

### F-013：考勤统计报表
- **用户故事**：作为门店店长，我需要在月底生成本店员工的考勤统计报表，以便核算工资和了解员工出勤情况
- **Kano 类型**：期望型
- **核心逻辑**：按月汇总每名员工的出勤天数、迟到次数、早退次数、缺卡次数；支持导出为 Excel
- **验收标准**：
  - [ ] 选择月份后，本店所有员工考勤汇总加载时间 ≤ 3 秒
  - [ ] 考勤报表可导出为 .xlsx 文件
  - [ ] 统计数据与考勤明细记录完全一致，零误差
- **依赖**：F-010、F-012

### F-015：总部广播公告
- **用户故事**：作为总部管理人员，我需要向所有门店发布通知公告，以便重要信息准确触达每家门店，不再依赖微信群
- **Kano 类型**：期望型
- **核心逻辑**：总部管理员发布公告（标题 + 正文 + 可选附图）；全部门店可见；按时间倒序展示；员工端收到 App 推送通知
- **验收标准**：
  - [ ] 总部发布公告后，15 分钟内所有门店员工可在 App 看到
  - [ ] 公告列表显示已读 / 未读状态（员工视角）
  - [ ] 历史公告可查找，保留时间不少于 1 年
- **依赖**：F-001、F-002

### F-017：微信小程序端
- **用户故事**：作为任意角色用户，我需要通过微信小程序访问系统核心功能，以便无需安装额外 App 即可完成日常操作（打卡、查排班、查看公告等）
- **Kano 类型**：期望型（综合评分 3.9）
- **核心逻辑**：微信小程序作为主要移动端载体，覆盖员工打卡（F-012）、排班查看（F-011）、公告通知（F-015）；店长可在小程序内录入销售数据（F-003）和查看本店看板（F-004）；总部管理人员 PC 端为主，小程序为辅；支持微信登录（解决 F-001 移动端认证）
- **验收标准**：
  - [ ] 员工通过微信扫码或搜索小程序即可访问，无需下载安装
  - [ ] 小程序内完成打卡操作不超过 2 步
  - [ ] 小程序加载首页时间 ≤ 3 秒（4G 网络）
  - [ ] 微信登录后与系统账号自动关联，无需重复输入账号密码
- **依赖**：F-001、F-002、F-011、F-012、F-015
- **注**：需决策小程序是否为唯一移动端（替代原生 App），或两者并存，见 Q-004；若确认小程序路线，F-012 打卡可使用微信位置 API（部分解决 Q-001）

## 可选功能（P2）

### F-014：代打卡（补录考勤）
- **用户故事**：作为门店店长，我需要在员工忘记打卡时帮其补录考勤记录，以便考勤数据完整，避免影响员工薪资核算
- **Kano 类型**：期望型
- **核心逻辑**：店长可为本店员工补录指定日期的打卡时间；补录操作全程记录操作日志（操作人、操作时间、被操作员工、补录日期）；补录记录在考勤明细中有"补录"标识；补录时间范围和是否需要审批待确认（见 Q-002、Q-003）
- **验收标准**：
  - [ ] 店长为指定员工补录考勤，操作步骤 ≤ 4 步
  - [ ] 每条补录记录自动生成操作日志，含：操作人、操作时间、目标员工、补录的考勤日期
  - [ ] 补录记录在考勤明细中有"补录"标识，区别于员工自主打卡
- **依赖**：F-010、F-012
- **注**：需确认补录时间范围和是否需要审批（Q-002、Q-003），设计时须考虑合规风险，防止滥用

### F-018：美团外卖订单自动同步
- **用户故事**：作为门店店长，我需要美团外卖订单数据自动同步到系统，以便销售数据更准确，减少手动录入工作量
- **Kano 类型**：期望型（综合评分 2.6）
- **核心逻辑**：通过美团开放平台 API 获取外卖订单数据；每日自动汇总外卖营业额并合并到 F-003 的销售数据中；区分堂食收入和外卖收入分别展示
- **验收标准**：
  - [ ] 外卖订单每日自动同步，延迟不超过 1 小时
  - [ ] 销售看板中堂食与外卖收入分列展示，合计正确
  - [ ] 同步失败时系统提醒店长手动补录，不静默丢失数据
- **依赖**：F-003、F-004
- **注**：前提条件：客户需在美团开放平台完成商家入驻和 API 授权，见 Q-006；15 家门店需逐一授权，有一定周期

### F-019：按财务模板导出报表
- **用户故事**：作为财务人员，我需要系统直接按财务部门固定模板格式导出报表，以便无需每次手动调整格式即可使用
- **Kano 类型**：期望型（综合评分 2.4）
- **核心逻辑**：在 F-006 现有导出基础上，新增"按财务模板导出"选项；模板格式由财务提供（字段顺序、汇总行、分类口径等）；生成 .xlsx 文件，直接符合财务使用习惯
- **验收标准**：
  - [ ] 导出文件的表头、字段顺序、合计行格式与财务提供的模板完全一致
  - [ ] 选择时间范围后，模板导出在 15 秒内完成
  - [ ] 同时保留原有通用 Excel 导出功能，不影响其他用户
- **依赖**：F-006
- **注**：⚠️ 前置条件：需要财务提供现有报表模板文件，见 Q-005；没有模板文件则无法实现本功能

### F-020：差评预警（美团/大众点评）
- **用户故事**：作为总部运营人员，我需要在美团或大众点评出现新差评时及时收到系统提醒，以便快速响应并处理客户投诉
- **Kano 类型**：兴奋型（综合评分 2.3）
- **核心逻辑**：通过美团/大众点评开放平台 API 定时拉取评价数据；评分低于阈值（如 ≤ 3 星）或含特定关键词时触发预警；推送通知到总部管理员和对应门店店长
- **验收标准**：
  - [ ] 新差评出现后，相关负责人在 30 分钟内收到预警通知
  - [ ] 预警通知包含：门店名称、评价平台、评分、评价内容摘要
  - [ ] 差评阈值（评分下限/关键词）可由总部管理员配置
- **依赖**：F-001、F-002
- **注**：依赖美团和大众点评分别开放 API 授权，见 Q-006；两个平台授权流程独立，可分步实现

### F-021：跨门店排班
- **用户故事**：作为门店店长，我需要将兼职于多个门店的员工排班到不同门店的班次，以便跨店员工能在各自班次正常打卡和出勤
- **Kano 类型**：期望型（综合评分 2.0）
- **核心逻辑**：员工档案支持关联多个门店（M:N 关系）；排班时可选择目标门店；系统自动检测时间冲突（同一员工同一时段不可在两家门店排班）；考勤按门店分开统计
- **验收标准**：
  - [ ] 员工可被关联到多个门店，关联关系由总部管理员维护
  - [ ] 为跨门店员工排班时，必须选择目标门店，系统检测时间冲突并给出提示
  - [ ] 跨门店员工的考勤报表按门店分别汇总，不混淆
- **依赖**：F-010、F-011、F-012
- **注**：⚠️ 数据模型影响：员工与门店的关系需从 1:1 升级为 M:N（一名员工可属于多个门店）。**强烈建议在 v1.0 设计 F-010 数据模型时提前预留此扩展**，避免 v1.1 上线时发生数据迁移成本。

### F-016：定向公告（按门店发送）
- **用户故事**：作为总部管理人员，我需要向特定门店发送专属通知，以便精准传达只与该门店相关的信息
- **Kano 类型**：兴奋型
- **核心逻辑**：发布公告时可选择接收范围：全部门店或指定门店（多选）；非目标门店的人员不可见该公告
- **验收标准**：
  - [ ] 发布公告时可按门店多选接收范围（默认选"全部"）
  - [ ] 目标门店外的员工无法看到该条定向公告
  - [ ] 定向公告与广播公告在列表中有清晰的视觉区分
- **依赖**：F-015

## 暂不考虑的功能

| 功能描述 | 暂不做的原因 |
|----------|--------------|
| 与收银系统 / POS 数据自动同步 | 客户未提及；系统集成复杂度和成本高；当前阶段手动录入已可满足需求 |
| 高级 BI 报表（同环比、预测分析、漏斗） | 超出当前版本范围；基础看板稳定后可作为 v2.0 增强方向 |
| 供应链 / 采购管理 | 超出客户初期定义的"门店运营管理"范畴 |
| 薪资自动核算与发放 | 考勤系统稳定后可作为后续扩展；初期仅做统计，薪资核算由客户自行处理 |
| 会员积分系统（消费积分兑换） | 属于 C 端客户运营功能，超出当前 B 端门店运营管理的产品定位；建议作为独立产品或 v2.0 规划 |
| 对接用友财务软件 | 用友 API 版本多（U8/U9/畅捷通），集成成本极高；受益人群仅财务部门（约 2~3 人）；投入产出比低，建议中长期规划 |
