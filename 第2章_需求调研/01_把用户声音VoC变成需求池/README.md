# pm-requirements · 课程录制参考脚本

> 面向 1-3 年产品经理的 AI 效率课程  
> **Skill 类型**：双模式 Workflow | **前置依赖**：无 | **输出格式**：.md / .docx / .xlsx

---

## 01 Skill 概览

这个 Skill 解决产品经理最头痛的问题之一：**怎么把乱七八糟的零散想法、客户邮件、会议纪要，变成一份清晰的功能清单和需求池。**

更像一个「需求管家」——不止做一次性的整理，还管后续不断进来的新需求，帮你判断是加进当前版本还是推到下一版。

**工作流程：**

```
用户输入（文档/聊天记录/碎片想法）
    → 自动检测（_pm_context/requirements/feature-list.md 是否存在？）
    → Mode A（初始化：整理 → 输出）  或  Mode B（增量：评估 → 更新）
    → 输出功能清单 + 需求池
```

### 两种工作模式

| 模式 | 触发条件 | 核心产出 |
|------|----------|----------|
| **Mode A：初始化** | 第一次打开，还没建立功能清单 | feature-list.md + requirement-pool.md |
| **Mode B：增量更新** | 功能清单已存在，用户又提新需求 | 更新功能清单 + 追加需求池记录 |

### 输出文件清单（自动生成）

| 文件名 | 说明 |
|--------|------|
| `feature-list.md` | AI 源文件 · 英文命名 · 始终最新版 |
| `requirement-pool.md` | 需求池源文件 · 持续追加不分版本 |
| `功能清单_vX.X.docx` | Word 版 · 宋体 · 有封面和概览表 |
| `功能概览表_vX.X.xlsx` | Excel 概览 · 版本迭代保留旧版 |
| `需求池.xlsx` | Excel 需求池 · 覆盖式更新 |

## 02 录制准备工作

> ⚠️ **演示前需要确认**
> - 电脑已登录 Claude Code（当前工作目录任意）
> - pm-requirements Skill 已安装（在 `.claude/skills/pm-requirements/` 目录下）

整个课程录制约 **10–15 分钟**，分三部分：

1. **前半段（约 3 分钟）：** 演示 Mode A，输入客户文档 + 口头想法，看 AI 怎么整理输出
2. **中间展示（约 2 分钟）：** 浏览生成的 .md 文件，看功能清单和需求池表格
3. **后半段（约 5 分钟）：** 演示 Mode B，输入新一轮新需求，看冲突检测和版本升级

## 03 录制脚本（分镜）

### PART 1 · 第一次输入 → Mode A 初始化（0:00 – 5:00）

**🎙 录制话术：**

> 我们来看一个真实场景。假设你是一家火锅连锁总部做产品的，客户刚发了一个需求文档过来，同时又口头交代了一些想法。我们就把这些材料一起扔给 AI 看看它怎么处理。

**输入内容：**

```
/pm-requirements

客户发来的需求文档在这里：D:\work\AI\AI PM Course Prod\test-input\client-requirements.md

另外昨天和张总电话聊，他口头补充了这些：
- 想加一个会员积分系统，客人消费可以积分兑换
- 能不能对接美团，让外卖订单自动同步进来
- 我们员工文化程度不高，系统要非常简单，最好像微信一样好上手
- 老板随时想用手机看数据，最好有个微信小程序
- 对了，之前财务说想把这个系统和用友财务软件打通
```

> 💡 **主讲提示：** 输入这段时，可以强调「可以看到这里既有结构化文档，也有零碎的几句话」，说明真实场景就是这样混合的。

AI 的 Step 1-3（解析→归类→排序）过程约 30 秒，然后输出 .md 文件。

**🎙 录制话术：**

> 好，AI 已经分析完了。它自动检测到我们还没有功能清单，所以进入了「初始化模式」——也就是从零开始帮你搭建需求体系。它现在会问我们确认功能清单的内容。我们点确认，让它继续生成 Word 和 Excel。

**在对话中看到 AI 展示的摘要后，输入确认：**

```
确认，没问题，继续生成吧
```

### PART 2 · 浏览输出文件（5:00 – 7:00）

**🎙 录制话术：**

> 好，现在输出完成了。我们来看看它生成了哪些文件。

打开输出文件夹：`_pm_context/requirements/`

| 文件名 | 内容 | 主讲要点 |
|--------|------|----------|
| `feature-list.md` | 功能清单源文件 | 有变更记录表、功能概览表、每个功能的用户故事和验收标准 |
| `requirement-pool.md` | 需求池源文件 | 10 条需求，包括已接受和推迟的，提出人全部有标注 |
| `功能清单_v1.0.docx` | Word 版 | 打开展示：封面、概览表、详细功能说明——团队日常看的版本 |
| `功能概览表_v1.0.xlsx` | Excel 概览 | 展示两个 Sheet：功能概览 + 需求池 |
| `需求池.xlsx` | Excel 需求池 | 展示持续追加的流水记录 |

> 💡 **主讲提示（重点讲这 3 点）：**
> 1. **冲突检测：**「做小程序」和「手机端为主」两者有重叠但不矛盾，AI 建议小程序作为移动端方案之一
> 2. **范围警告：** 会员积分、用友对接、美团同步被识别为「超出 v1.0 核心范围」，标记推迟到 v2.0
> 3. **隐含需求补全：** AI 自动补上了「用户与登录」「角色权限管理」这两个文档里没写但必有的功能

### PART 3 · 新需求来了 → Mode B 增量更新（7:00 – 12:00）

**🎙 录制话术：**

> 功能清单做好之后，过了一周，销售说客户开会又提了新需求。我们直接输入看看会发生什么。

**第二次输入（新对话窗口）：**

```
/pm-requirements

销售说客户那边刚开了个会，又提了几个新需求：

1. 张总说要加一个"代打卡"功能，店长可以帮员工补录昨天忘打的卡

2. 财务那边说，光导出 Excel 不够，她们用的报表格式是固定的，
   能不能直接按她们的模板格式导出，不用她们再手动调格式

3. 运营提了，希望总部能看到各门店的"差评预警"，
   就是美团/大众点评上如果有新差评，系统能自动提醒

4. 张总还说，排班那块，现在一个员工只能排在一个门店，
   但他们有几个员工是跑两个门店的，希望能支持跨门店排班
```

> ⚠️ **注意：** 这里要**开新对话窗口**，因为 Claude 只记得当前对话的内容。但 AI 会检测到桌面上已有功能清单文件，自动进入 Mode B 增量更新模式。

**🎙 录制话术：**

> 你看，AI 没有重新整理，而是直接读出现有功能清单，逐条分析新需求跟已有功能的关系。它发现代打卡和考勤打卡是合并关系、模板导出和数据导出是升级关系、跨门店排班会改动排班和员工两个模块的数据模型……甚至指出差评预警依赖美团 API，而美团对接本身还是待确认状态。这些分析比我们自己拍脑袋要全面得多。

| 新需求 | AI 的分析结果 | 处置 |
|--------|---------------|------|
| 代打卡 | 和 F-008 考勤打卡直接相关，需注意数据真实性问题 | **合并**到 F-008 |
| 按模板导出 | 和 F-011 数据导出高度重叠，可以升级为「数据导出与报表」 | **合并**到 F-011 |
| 差评预警 | 新功能，但依赖 F-013 美团 API 的状态 | **新增** F-016，归入 v1.1 |
| 跨门店排班 | F-003 原设计一对一，改动波及 F-004 员工数据模型 | **修改** F-003 和 F-004 |

**确认更新：**

```
确认，没问题
```

**🎙 录制话术：**

> 可以看到版本号从 v1.0 变成了 v1.1，变更记录里清清楚楚写明了改了哪些。旧版 v1.0 的文件也在，没有覆盖。需求池里多了 4 条新需求，每条都有提出人、处置结果和备注。这就是一个持续维护的需求体系该有的样子。

### 总结：这个 Skill 的几个关键设计

1. **写项目符号 vs 写用户故事** —— 每个功能强制写「作为 X，我需要 Y，以便 Z」，而不是只写功能名
2. **冲突不藏着** —— 发现需求矛盾直接标红，请 PM 决策，不偷偷两个都收了
3. **拒绝的也要记录** —— 需求池记录所有进入评估的需求，防止下周同一个需求又被提出来
4. **版本有痕** —— 每次变更有版本号留底，旧版 Word/Excel 保留不删除
5. **AI 建议，PM 决策** —— Skill 只给出分析和建议，最终版本归属和接受/拒绝由人决定

## 04 测试用例（供录制使用）

### 第一轮输入（Mode A 初始化）

**场景文档**（`client-requirements.md`）内容如下：

```
# 运营管理系统需求文档

客户：某连锁火锅品牌（15家门店）
对接人：市场部张总
日期：2026年5月

## 背景说明

我们现在门店管理全靠微信群和Excel，每天各门店店长要手动汇报当天营业额，
总部统计很费事，经常出错。库存也是各门店自己记，总部不清楚情况，
有几次都断货了。希望做一套系统，把门店管理这块统一起来。

## 核心需求

### 1. 销售数据管理
- 各门店每天能录入当天的营业额、客单价、翻台率
- 总部能看到所有门店的汇总数据
- 支持按时间段查看趋势（周/月/季度）
- 数据能导出 Excel

### 2. 库存管理
- 各门店能记录每天的食材入库、消耗、库存余量
- 当库存低于预警线时，系统自动提醒
- 总部可以查看各门店库存情况

### 3. 员工管理
- 录入员工基本信息（姓名、岗位、入职时间）
- 排班功能：店长可以在系统里排班，员工可以看到自己的班次
- 考勤记录：员工打卡，系统自动统计出勤

### 4. 公告通知
- 总部可以发布通知公告，所有门店都能看到
- 支持按门店定向发送

## 使用人群

- 总部管理人员：看数据、发通知、管理权限（约5人）
- 门店店长：录数据、排班、看本店报表（15人，每店1人）
- 普通员工：看排班、打卡（约300人）

## 其他说明

系统主要在手机上用，电脑端也要有。预算有限，希望先把最核心的做出来，
其他功能后面再加。
```

**口头补充（由主讲人在对话中追加）：**

```
另外昨天和张总电话聊，他口头补充了这些：
- 想加一个会员积分系统，客人消费可以积分兑换
- 能不能对接美团，让外卖订单自动同步进来
- 我们员工文化程度不高，系统要非常简单，最好像微信一样好上手
- 老板随时想用手机看数据，最好有个微信小程序
- 对了，之前财务说想把这个系统和用友财务软件打通
```

### 第二轮输入（Mode B 增量更新）

在全新的对话窗口中输入，AI 会自动检测到已有 `feature-list.md`，进入 Mode B。

## 05 录制中可能遇到的问题

| 问题 | 原因 | 处理方式 |
|------|------|----------|
| AI 没有进入 Mode B | 检测不到已有 feature-list.md，可能因为路径或工作目录变了 | 检查 `_pm_context/requirements/` 是否存在；或手动告诉 AI「我已有功能清单，帮我处理新需求」 |
| 生成的 Word/Excel 是英文 | docx/xlsx skill 版本差异 | 不影响内容，课程中可提一句「输出语言可配置」 |
| AI 要求安装 python-docx 或 openpyxl | 当前环境没有对应库 | 允许安装即可，或在课前提前装好 `pip install python-docx openpyxl` |
| Word 字体不是宋体 | 生成脚本对字体的指定依赖运行环境 | 手动修改字体不影响课程演示效果，可正常讲解内容 |
