# /brainstorm -- 多维度构思 (Multi-Perspective Ideation)

从三个视角（产品经理、设计师、工程师）生成创意产品构思或实验设计，并根据你是处理现有产品还是构建新产品进行定制。

## 调用示例

```
/brainstorm ideas existing 提高移动银行应用的参与度
/brainstorm ideas new 为忙碌的父母提供 AI 驱动的饮食计划
/brainstorm experiments existing 重新设计新手引导流程
/brainstorm experiments new 自由设计师的平台/市场
/brainstorm                          # 交互模式 —— 询问你的需求
```

## 工作流程

### 第 1 步：确定模式

解析参数以识别两个维度：

1. **构思内容**：`ideas`（功能概念）或 `experiments`（验证测试）
2. **产品阶段**：`existing`（持续探索）或 `new`（初始探索）

如果缺少任一维度，请询问用户。如果两个都缺少，请询问：
- "你是在集思广益 **构思 (ideas)** 要构建的内容，还是 **实验 (experiments)** 来验证假设？"
- "这是针对 **现有 (existing)** 产品还是 **新 (new)** 产品概念？"

### 第 2 步：收集背景

向用户询问背景信息。采取对话式 —— 首先询问最关键的问题：

**针对现有产品：**
- 产品是什么？当前用户是谁？
- 你正在探索哪个机会领域或问题空间？
- 有哪些约束条件（技术债、平台限制、团队能力）？
- 之前尝试过什么？

**针对新产品：**
- 产品概念是什么？它解决什么问题？
- 目标用户是谁？他们目前的替代方案是什么？
- 你处于什么阶段？（草图、已验证的问题、早期原型）
- 最具风险的假设是什么？

接受来自上传文件（PRD、研究文档、战略幻灯片）、粘贴文本或对话的背景信息。

### 第 3 步：生成输出

**如果是集思广益构思 (ideas)** —— 应用 **brainstorm-ideas-existing** 或 **brainstorm-ideas-new** 技能：
- 从三个视角生成构思：产品经理（用户价值、业务影响）、设计师（UX、惊喜感、易用性）、工程师（技术创新、平台利用、可扩展性）
- 为每个构思提供：名称、描述、目标用户影响、可行性评估
- 排出前 5 个构思并说明理由
- 标记哪些构思可能是“快速见效”或“战略博弈”

**如果是集思广益实验 (experiments)** —— 应用 **brainstorm-experiments-existing** 或 **brainstorm-experiments-new** 技能：
- 针对现有产品：建议 A/B 测试、原型、虚假门测试、奥兹国大王、礼宾模式实验和技术探针 (Spikes)
- 针对新产品：创建 XYZ+S 假设并建议原型 (Pretotype) 实验（落地页、讲解视频、预订、礼宾模式 MVP）
- 为每个实验提供：假设、方法、成功标准、投入估计、预期时间线
- 按“学习/投入比”进行排名

### 第 4 步：深入迭代

在展示初步结果后，提供：
- "是否需要我将其中任何构思 **细化** 为完整的规范？"
- "我是否应该 **识别** 核心构思背后的 **假设**？"（链接到 `identify-assumptions-existing` 或 `identify-assumptions-new` 技能）
- "是否想要为核心构思 **设计实验** 进行验证？"（链接到实验模式）
- "我是否应该根据你当前的待办列表对这些内容进行 **优先级排序**？"（链接到 `prioritize-features` 技能）
