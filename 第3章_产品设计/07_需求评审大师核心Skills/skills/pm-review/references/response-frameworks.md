# 应对话术框架（Response Frameworks）

> 供 pm-review Skill 在 Step 4 生成应对方案时引用。包含通用应对框架和场景化话术。

---

## 核心原则

1. **不防御，先认可**：先肯定对方提了一个好问题，再展开回应
2. **结构化回应**：背景 → 分析 → 方案 → 下一步，逻辑清晰
3. **用数据说话**：能用数据回应的场景，不用主观判断
4. **有边界意识**：PM 负责方向和范围，技术细节留给技术评审
5. **留后手**：所有回应都给出下一步行动，不在会上空对空讨论

---

## 通用应对框架

### 框架 1：PREP 模型（观点-理由-例子-观点）

适用于需要说服对方接受某个方案时。

```
P（Point 观点）：我认为我们应该先做[A]而不是[B]。
R（Reason 理由）：因为[A]能覆盖 80% 的核心场景，而且开发成本只有[B]的 1/3。
E（Example 例子）：参考[X产品]的做法，他们也是先做简化版再逐步迭代。
P（Point 重申）：所以建议先把[A]做扎实，[B]放到下个迭代。
```

### 框架 2：SPIN 回应（场景-问题-影响-需求）

适用于被质疑功能价值时。

```
S（Situation 场景）：目前的场景是[描述现状]。
P（Problem 问题）：用户在这个过程中遇到的痛点是[具体问题]。
I（Implication 影响）：如果不解决，会导致[业务影响，最好有数据]。
N（Need-payoff 需求收益）：解决后可以带来[预期收益]。
```

### 框架 3：3C 决策框架（Context-Comparison-Choice）

适用于多个方案需要选择时。

```
Context（背景）：我们面临的选择是[A]还是[B]。
Comparison（对比）：A 的优势是[X]，风险是[Y]；B 的优势是[Z]，风险是[W]。
Choice（选择）：综合来看，我建议选[A]，理由是[一句话]。
```

## 场景化话术

### 场景：被问到没准备的问题

> "这个点提得很好，我之前确实没有在这一版 PRD 里展开。让我从[已有信息]的角度先回应一部分——目前我们的初步判断是[方向和理由]。具体的技术方案需要在设计阶段和[角色]一起敲定。我记一下，会后和[角色]单独对齐，下次同步。"

**关键动作：** 认可 → 给方向 → 承诺跟进 → 记下来（真的记）

### 场景：技术质疑"这个很难实现"

> "理解。那我们来看一下，核心目标是不变的，但实现路径可以调整。你觉得如果先做到[简化版本]，最难的技术点[具体点]是否可以规避？或者有没有你之前做过的类似方案可以参考？"

**关键动作：** 不争论难度 → 聚焦目标 → 探索替代路径

### 场景：多个角色意见冲突

> "我理解[A角色]关注[某方面]，[B角色]关注[另一方面]，两个角度都很重要。今天我们评审的核心目的是确认需求方向是否正确，具体实现上的取舍我建议在技术评审会上专门讨论。作为 PM，我的初步建议是[折中方向]，核心原则是[决策依据]。大家觉得这个原则可以吗？"

**关键动作：** 不站队 → 确认共同目标 → 提出决策原则 → 升维到原则讨论

### 场景：老板临时增加需求（范围蔓延）

> "这个方向很有价值，我先记下来。不过在当前版本范围里加入，可能会影响[A]和[B]的交付时间，这两项是[X业务方]的硬需求。我建议我们先把这个版本的核心功能交付，[新需求]作为下个迭代的第一优先。您觉得这样的节奏安排可以吗？或者如果这个需求必须跟本版本一起上，我们可能需要讨论哪些功能可以往后移。"

**关键动作：** 不说不做 → 陈述影响 → 给选择 → 让老板决策

### 场景：被质疑"这个需求的 ROI 够吗"

> "好问题。目前我们判断的依据是[数据A]和[数据B]，预估对[核心指标]的影响是[X%]。如果大家觉得数据支撑不够充分，我们可以先做一个极简版快速验证——用最小的开发成本跑两周数据，再决定是否全量投入。这样风险和投入都可控。"

**关键动作：** 展示已有的 → 承认不确定性 → 提出低风险验证方案

### 场景：评审陷入细节纠缠

> "这个问题我建议我们在技术评审会上专门讨论，今天先确认大方向。我关心的是——这个需求的方向大家认不认可？如果方向没问题，实现层面的细节我们单独拉会解决。"

**关键动作：** 升维 → 拉回到核心议题 → 给细节问题安排出口

## 按角色的话术速查

| 角色 | 被挑战时的锚点 | 一句话回应 |
|------|---------------|-----------|
| 前端 | "实现复杂" | "我们先做核心路径，降级方案用[简单方案]，后续迭代再优化体验。" |
| 后端 | "数据模型不支持" | "会后我们单独对一下数据模型，把[具体问题]梳理清楚，下次评审同步。" |
| 设计 | "不符合设计规范" | "我们参考了[竞品/数据]，会后整理对比分析，看是否需要和设计规范对齐。" |
| 测试 | "测试场景太多" | "[核心场景]是必须覆盖的，[边界场景]优先级可以降一级，测试同学帮忙评估。" |
| 业务方 | "这功能没人用" | "我们和[X]团队确认过，这个功能直接影响[指标]。上线后追踪[数据]验证。" |
| 老板 | "为什么现在做" | "现在做的原因是[时机/竞争/用户压力]，不做的话可能[后果]。" |

## 评审后跟进动作

评审结束后，PM 应完成以下跟进：

1. **问题台账**：记录所有未解决问题，标注责任人和截止时间
2. **需求更新**：根据评审结果更新 PRD 或原型
3. **纪要分发**：24 小时内发送评审纪要给所有参会人
4. **单独对齐**：对会上无法快速达成一致的问题，会后 1v1 沟通
5. **下次评审**：明确下次评审的时间和议题
