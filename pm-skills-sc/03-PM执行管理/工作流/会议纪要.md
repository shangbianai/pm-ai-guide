# /meeting-notes -- 会议总结

将原始会议记录或草稿转换为清晰、结构化的会议纪要，捕获决策并分配行动项。

## 调用示例

```
/meeting-notes [粘贴会议记录]
/meeting-notes [上传会议记录文件、音频总结或笔记]
```

## 工作流程

### 第 1 步：接收记录

接受任何格式：
- 完整转录文本（来自 Otter、Fireflies、Google Meet、Zoom 等）
- 会议期间记录的草稿笔记
- 录音工具生成的音频总结或会议简报
- 多个输入（例如：转录文本 + 用户自己的笔记）

如果输入内容稀疏，请基于现有信息进行处理并标记缺失项。

### 第 2 步：提取与结构化

应用 **summarize-meeting** 技能：

解析内容以识别：
- **参会者**: 谁出席了会议（通过介绍、发言人标签或提及）
- **讨论话题**: 主要议程项目或对话主线
- **达成的决策**: 明确的协议或达成的结论
- **行动项**: 分配的任务，包含负责人和截止日期（如果提及）
- **待解决问题**: 需要后续跟进的未决事项
- **关键语录**: 值得逐字保留的重要陈述
- **背景信息**: 会议类型、项目及背景

### 第 3 步：生成会议总结

```
## 会议总结

**日期**: [如果已知则填写日期]
**参会者**: [姓名/角色]
**会议类型**: [站会、计划会、评审会、1:1、利益相关者会议等]
**主题**: [主要议题]

### 概要
[3-5 句概述讨论内容及结论]

### 关键决策
1. **[决策]** — [背景和理由]
2. ...

### 行动项
| # | 任务 | 负责人 | 截止日期 | 状态 |
|---|--------|-------|----------|--------|

### 讨论亮点
**[话题 1]**: [关键点、不同观点、结论]
**[话题 2]**: [关键点、不同观点、结论]

### 待解决问题
- [问题] — 需要 [人员/团队] 的输入

### 后续步骤
- [接下来的安排]
- 下次会议: [如果提及]
```

保存为 markdown 文件。

### 第 4 步：后续建议

- "需要我将这些 **笔记通过电子邮件发送** 给参会者吗？"
- "我是否应该根据行动项 **创建任务工单 (Tickets)**？"
- "需要我根据达成的决策 **起草一份利益相关者更新** 吗？"

## 注意事项

- 决策是最有价值的输出 —— 确保每个决策都被清晰捕获
- 没有负责人的行动项是无效的 —— 如果未提及负责人，请予以标记
- 保持总结简洁 —— 没参加会议的人应该能在 30 秒内了解要点
- 如果转录文本非常长（超过 60 分钟的会议），请在完整总结前提供一个 TL;DR（太长不看版）
- 区分 "讨论过" 和 "已决定" —— 许多话题会被探讨但未必能达成结论
