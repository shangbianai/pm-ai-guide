```markdown
## 应对话术库

### 通用应对框架

#### 场景 1：被问到没准备的问题
> "这是一个很好的切入点，让我从[X]的角度来补充。目前我们的初步考虑是[Y]，具体方案需要在设计阶段和[角色]一起细化。我在会后和[角色]单独对齐，下次评审同步进展。"

#### 场景 2：多个角色意见冲突
> "我理解[A角色]关注[X]，[B角色]关注[Y]，这两个角度都很重要。我们今天的重点是确认需求方向是否正确，具体实现方案可以在技术评审会上专门讨论。我建议[折中方案或决策原则]。"

#### 场景 3：需求被质疑价值
> "这个需求的背景是[数据/用户反馈/业务目标]。我们预估上线后[X]指标会有[Y]的提升。如果大家觉得数据不够充分，我们可以先做一个最小版本快速验证，再决定是否全量投入。"

#### 场景 4：老板临时增加需求
> "这个方向很有价值，我先记下来。不过我担心在当前版本范围里加入会影响到[A]和[B]的交付时间。建议我们先把这个版本的核心功能交付，[新需求]可以作为下个迭代的优先项。您觉得这样的节奏可以吗？"

### 角色专用话术

| 角色 | 场景 | 话术 |
|------|------|------|
| 前端 | 质疑交互复杂度 | "我理解这个交互实现成本不低。我们可以先做核心路径，降级方案用[简单方案]，后续迭代再优化体验。" |
| 后端 | 质疑数据模型 | "会后我和后端同学单独对一下数据模型，把[具体问题]梳理清楚，下次评审同步。" |
| 设计 | 质疑交互规范 | "这个交互我们参考了[竞品/规范]，会后我整理一个对比分析，看看是否要和设计规范对齐。" |
| 测试 | 质疑测试覆盖 | "[X]场景确实需要重点测试，我会在验收标准里明确测试用例，测试同学帮忙补充边界情况。" |
| 业务方 | 质疑功能价值 | "我们和[X]业务团队确认过，这个功能直接影响[指标]。上线后我们会追踪[数据]来验证效果。" |
```

## 模式 C：一致性评审报告

模式 C 的报告在模式 B 基础上增加两个章节，其余相同。

### 增加的章节 1：一致性问题（放在「需求摘要」之后、「角色评审详情」之前）

```markdown
## 一致性问题

| 编号 | 比对维度 | 不一致描述 | 类型 | 严重度 |
|------|----------|------------|------|--------|
| C-001 | 功能覆盖 | PRD 有 [X] 功能但原型中未找到对应页面 | 原型遗漏 | 🔴 |
| C-002 | 字段对齐 | 原型展示了 [X] 字段但 PRD 未定义 | PRD 遗漏 | 🟡 |
| C-003 | 流程一致 | PRD 的操作顺序是 A→B→C，但原型的跳转是 A→C→B | 描述冲突 | 🔴 |

> 类型说明：
> - 「PRD 遗漏」= PRD 需要补充，原型是对的
> - 「原型遗漏」= 原型需要补充，PRD 是对的
> - 「描述冲突」= 两边不一致，需要对齐确认
```

### 增加的章节 2：原型解析摘要（放在「一致性问题」之后）

```markdown
## 原型解析摘要

[同模式 A 的「原型解析摘要」格式：页面清单 + 四态覆盖检查表]
```

### 增加的章节 3：交互自查结果（放在「角色评审详情」之后、「应对话术库」之前）

```markdown

[同模式 A 的「交互自查结果」格式：四态问题 + 表单问题 + 导航问题 + 数据展示问题]
```
