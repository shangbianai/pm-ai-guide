# /plan-okrs -- 团队 OKR 计划

生成结构良好的 OKR，将团队工作与公司战略联系起来。产出 3 组包含定性目标 (Objectives) 和定量关键结果 (Key Results) 的 OKR。

## 调用示例

```
/plan-okrs 增长团队 Q2 —— 公司目标是 ARR 增长 50%
/plan-okrs 针对 "提高激活率" 的入职小组 OKR
/plan-okrs [上传公司 OKR 或战略文档]
```

## 工作流程

### 第 1 步：收集背景信息

询问用户：
- 这些 OKR 是为哪个团队或产品领域制定的？
- 时间周期是多久？（季度是标准，也可以是年度或自定义）
- 这些 OKR 应该挂钩的公司级目标是什么？
- 上个季度发生了什么？（达成情况、未达标项、经验教训）
- 是否有任何限制或已知的优先级？

接受上传的公司 OKR 或战略文档。

### 第 2 步：生成 OKR

应用 **brainstorm-okrs** 技能：

- 创建 3 组 OKR（每个目标包含 3-5 个关键结果）
- **目标 (Objectives)**: 定性的、鼓舞人心的、具有挑战性但可实现的、行动导向的
- **关键结果 (Key Results)**: 定量的、可衡量的、有时限的、有明确负责人
- 确保 OKR 与公司目标挂钩，并有清晰的逻辑连接
- 平衡领先指标（活动/过程）与滞后指标（结果/产出）

### 第 3 步：质量验证

对照最佳实践检查每个 OKR：
- 目标是否鼓舞人心？（你能以此团结团队吗？）
- 关键结果是否可衡量？（你能通过数据而非主观判断来检查完成情况吗？）
- 目标是否具有挑战性但不过于离谱？（达成 70% = 校准良好）
- 每个目标是否有 3-5 个关键结果？（过多会导致失去焦点）
- 关键结果是否避免了被 "刷数据" 的风险？（例如："交付 5 个功能" 会激励团队交付垃圾功能）

标记任何问题并提出改进建议。

### 第 4 步：呈现与迭代

```
## 团队 OKR: [团队名称] — [周期]

**挂钩目标**: [公司目标]

### 目标 1: [鼓舞人心的定性陈述]
| # | 关键结果 | 基准线 | 目标值 | 负责人 |
|---|-----------|----------|--------|-------|
| KR1 | [可衡量的结果] | [当前值] | [目标值] | [团队/个人] |
| KR2 | ... | ... | ... | ... |
| KR3 | ... | ... | ... | ... |

### 目标 2: [鼓舞人心的定性陈述]
[相同格式]

### 目标 3: [鼓舞人心的定性陈述]
[相同格式]

### 对齐地图
公司目标 → 团队目标 → 关键结果 → 预期影响

### 评分指南
- 0.0-0.3: 显著未达标 —— 需要调查并学习原因
- 0.4-0.6: 有进展但未达预期
- 0.7-0.9: 校准良好的挑战性目标 —— 这是理想的目标区域
- 1.0: 要么完美达成，要么目标设得不够具有挑战性

### 检查节奏
- **每周**: 对每个 KR 进行快速的红绿灯状态更新
- **季中**: 深度评估，如果背景发生变化则调整目标
- **季末**: 评分、反思，为下个季度提供参考
```

提供建议：
- "需要我 **调整挑战难度** —— 让它们更激进或更稳健吗？"
- "我是否应该 **创建一个指标仪表盘** 来跟踪这些 OKR？"
- "需要我 **起草一份利益相关者更新** 来介绍这些 OKR 吗？"

## 注意事项

- OKR 应描述产出结果 (Outcomes)，而非产出物 (Outputs)（是 "提高 20% 激活率"，而不是 "上线入职流程重构"）
- 如果用户没有公司 OKR，帮助他们从产品战略或业务目标中推导团队目标
- 每个团队每季度最多 3 个目标 —— 过多意味着失去重点
- 关键结果应该是挑战性目标 —— 如果你确定能 100% 完成，说明它们不够具有挑战性
- 标记任何可能被操纵的 KR，并建议一个对冲指标
