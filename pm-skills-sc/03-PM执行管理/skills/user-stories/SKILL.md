# 用户故事 (User Stories)

遵循 3C（卡片 Card、对话 Conversation、确认 Confirmation）原则和 INVEST 标准创建用户故事。生成包含描述、设计链接和验收标准的故事。

**使用场景：** 编写用户故事、将功能分解为故事、创建待办项 (Backlog items) 或定义验收标准。

**参数：**
- `$PRODUCT`: 产品或系统名称
- `$FEATURE`: 要分解为故事的新功能
- `$DESIGN`: 设计文件链接 (Figma, Miro 等)
- `$ASSUMPTIONS`: 关键假设或背景

## 逐步流程

1. **分析功能**：基于提供的设计和背景进行分析
2. **识别用户角色**：以及不同的用户旅程
3. **应用 3C 框架**：
   - 卡片 (Card)：简单的标题和一句话描述
   - 对话 (Conversation)：关于意图的详细讨论
   - 确认 (Confirmation)：清晰的验收标准
4. **遵守 INVEST 标准**：独立性 (Independent)、可协商性 (Negotiable)、有价值 (Valuable)、可估算 (Estimable)、短小 (Small)、可测试 (Testable)
5. **使用通俗语言**：小学毕业生也能理解的语言
6. **链接设计文件**：用于视觉参考
7. **输出用户故事**：以结构化格式输出

## 故事模板

**标题：** [功能名称]

**描述：** 作为一个 [用户角色]，我想要 [执行动作]，以便于 [获得收益]。

**设计：** [设计文件链接]

**验收标准：**
1. [清晰、可测试的标准]
2. [可观察的行为]
3. [系统正确验证]
4. [边界情况处理]
5. [性能或无障碍考量]
6. [集成点]

## 用户故事示例

**标题：** 最近查看部分

**描述：** 作为一个在线购物者，我希望在产品页面上看到“最近查看”部分，以便轻松重新访问我考虑过的商品。

**设计：** [Figma 链接]

**验收标准：**
1. 对于之前至少查看过 1 个产品的每位用户，在产品页面底部显示“最近查看”部分。
2. 对于在当前会话中访问第一个产品页面的用户，不显示该部分。
3. 当前产品本身不包含在显示的项目中。
4. 该部分展示带有图片、标题和价格的产品卡片或缩略图。
5. 每个产品卡片指示查看时间（例如，“5 分钟前查看”）。
6. 点击产品卡片会将用户引导至相应的产品页面。

## 输出交付物

- 该功能的完整用户故事集
- 每个故事包括标题、描述、设计链接和 4-6 条验收标准
- 故事相互独立，可以按任何顺序开发
- 故事大小适合一个 Sprint 周期
- 故事引用相关的设计文档
