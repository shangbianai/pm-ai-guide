# /triage-requests -- 功能需求筛选 (Feature Request Triage)

将一堆功能需求 —— 无论是来自支持工单、销售电话、调查还是 Slack —— 转化为优先级明确、可操作的待办列表。

## 调用示例

```
/triage-requests                           # 询问输入内容
/triage-requests [粘贴需求列表]
/triage-requests [上传 CSV/电子表格]
```

## 工作流程

### 第 1 步：接受功能需求

接受任何格式的输入：
- **粘贴的文本**：需求列表，每行或每段一个
- **上传的文件**：包含需求数据的 CSV、Excel 或文本文件
- **结构化数据**：如果输入包含列（请求者、需求、日期等），请予以保留

如果没有提供输入，请要求用户粘贴或上传他们的功能需求。

解析每个需求以提取：
- 核心诉求（用户想要什么）
- 背景信息（谁提出的、何时、为什么 —— 如果可用）
- 频率信号（有多少人提出了类似的需求）

### 第 2 步：收集优先级背景

向用户询问（采取对话式，不要一次性问完）：
- 产品是什么？它处于什么阶段？
- 目前的战略目标或 OKR 是什么？（有助于评估一致性）
- 有哪些约束条件需要考虑？（团队规模、技术债、即将到来的截止日期）
- 是否有某些细分市场的需求权重更高？（企业级客户、流失用户、核心用户）

### 第 3 步：分类与分析

应用 **analyze-feature-requests** 技能：

- **主题聚类**：将类似的需求归入主题（例如，“报告与分析”、“协作”、“移动端体验”）
- **每个主题的需求计数**：有多少个独特的需求对应每个主题
- **战略一致性**：对照所述目标对每个主题进行评分（高/中/低/无）
- **细分市场分析**：哪些用户细分市场在驱动哪些主题
- **情感信号**：需求是否伴随着挫折感、流失威胁或喜爱？

### 第 4 步：优先级排序

应用 **prioritize-features** 技能：

针对每个主题（以及每个主题内的核心需求）：

| 因素 | 评估 |
|------|------|
| **影响 (Impact)** | 影响多少用户？严重程度如何？ |
| **战略一致性** | 是否服务于当前目标？ |
| **投入估计** | T 恤尺码 (S/M/L/XL) |
| **风险** | 如果我们不做会发生什么？ |
| **收入信号** | 是否与交易、留存或扩张挂钩？ |

对主题进行排名并生成优先级列表。

### 第 5 步：生成筛选报告

```
## 功能需求筛选报告

**日期**: [今天]
**分析的需求数**: [计数]
**识别的主题数**: [计数]

### 主题摘要
| # | 主题 | 需求数 | 核心诉求 | 一致性 | 影响 | 投入 | 优先级 |
|---|------|--------|----------|--------|------|------|--------|

### 优先级 1：立即行动
[近期计划中应包含的主题/需求]
- **[主题]**: [X] 个需求 — [为什么它很紧急]
  - 核心需求: [列表]
  - 推荐行动: [构建 / 原型 / 调研]

### 优先级 2：后续计划
[值得计划但不紧急的主题]

### 优先级 3：收集更多信号
[有潜力但证据不足的主题]

### 优先级 4：拒绝或推迟
[与战略不符的需求 —— 说明理由]

### 值得注意的单项需求
[未聚类为主题但具有高价值的单项需求]

### 模式与洞察
- [关于用户诉求的关键洞察]
```
