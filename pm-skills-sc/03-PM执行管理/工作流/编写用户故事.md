# /write-stories -- 待办事项生成器

将功能拆解为结构良好的待办事项 (Backlog Items)。根据团队偏好从三种格式中选择，每种格式都包含完整的验收标准。

## 调用示例

```
/write-stories user 允许用户将报告导出为 PDF 和 CSV
/write-stories job 任务截止日期的通知系统
/write-stories wwa 移动应用的深色模式
/write-stories [上传 PRD 或功能规格书]      # 询问格式偏好
/write-stories                              # 询问功能和格式
```

## 格式说明

### 用户故事 (User Stories)
**格式**: "作为一个 [用户类型]，我想要 [能力]，以便 [获益]"
应用 **user-stories** 技能：
- 遵循 3C 原则：卡片 (Card - 简短描述)、对话 (Conversation - 背景)、确认 (Confirmation - 验收标准)
- 按照 INVEST 原则验证：独立性、可协商性、有价值、可估算、短小、可测试
- 在相关处包含设计链接或原型引用

### Job Stories
**格式**: "当 [场景] 时，我想要 [动机]，以便我可以 [结果]"
应用 **job-stories** 技能：
- 专注于场景和背景，而不是用户角色
- 立足于研究中观察到的真实用户场景
- 非常适合以 JTBD (Jobs-to-be-Done) 为导向的团队

### WWA (Why-What-Acceptance)
**格式**: 为什么 (Why) [战略背景] → 做什么 (What) [交付物] → 验收 (Acceptance) [标准]
应用 **wwas** 技能：
- 包含战略决策理由（我们为什么要构建这个）
- 产出独立、有价值、可测试的项目
- 适合需要业务背景的跨职能团队

## 工作流程

### 第 1 步：接收功能需求

接受任何形式：PRD、功能描述、用户研究发现或口头想法。如果提供了 PRD，则提取需求进行拆解。

### 第 2 步：确定格式

如果在调用时未指定，请询问：
- "你们团队使用哪种格式？**用户故事 (User Stories)**、**Job Stories** 还是 **WWA**？"
- 如果不确定，建议默认使用用户故事。

### 第 3 步：拆解功能

- 将功能拆解为 5-15 个独立的故事（足够小，可以在一个冲刺内完成）
- 确保每个故事都是独立有价值的（其自身就能交付用户价值）
- 按依赖关系和优先级排序
- 为每个故事编写 3-5 条验收标准
- 标记需要设计输入或技术调研 (Spikes) 的故事

### 第 4 步：生成故事

```
## 待办事项列表: [功能名称]

**格式**: [User Stories / Job Stories / WWA]
**总故事数**: [数量]
**预估总工作量**: [如果可能]

### 故事详情

#### 故事 1: [简短标题]
**[所选格式的故事陈述]**

验收标准:
- [ ] [标准 1]
- [ ] [标准 2]
- [ ] [标准 3]

优先级: [P0/P1/P2] | 工作量: [S/M/L] | 依赖项: [无或列表]

---
[为每个故事重复上述格式]

### 故事地图 (Story Map)
[视觉排序：必须有 → 应该有 → 可以有]

### 技术笔记
[横向关注点：API 变更、数据迁移、基础设施]

### 待解决问题
[在开始实施前需要回答的问题]
```

保存为 markdown 文件。

### 第 5 步：后续建议

- "需要我为这些故事 **生成测试场景** 吗？"
- "我是否应该为开发和测试 **创建模拟数据**？"
- "需要我为这些故事 **估算冲刺容量** 吗？"
- "我是否应该 **转换为不同的格式**（用户故事 ↔ Job Stories ↔ WWA）？"

## 注意事项

- 一个故事 = 一个可部署的价值单元 —— 如果它需要另一个故事才能发挥作用，则应合并
- 验收标准应能让 QA 在无需额外解读的情况下进行测试
- 错误处理和边界情况应有独立的故事，而不是作为正常路径故事下的子项
- 如果功能很大（超过 15 个故事），建议分组为史诗 (Epics) 或阶段
- 标记任何需要 Spike（在估算前进行技术调查）的故事
