# pm-compete · Skill 介绍与录制指引

> Claude Code Skill · AI PM 效率课  
> **竞品分析，从3天到30分钟**

输入竞品名称、链接或截图，自动生成功能矩阵、定位图、价值曲线与完整分析报告。方法论级输出，一条 prompt 触发。

| 指标 | 数值 |
|------|------|
| 支持输入形式 | 4 种 |
| 结构化输出项 | 7 项 |
| 分析方法论层次 | 5 层 |
| 工作模式 | 2 种 |

---

## Part 1 · Skill 介绍

### 适用场景

| # | 场景 | 说明 | 模式 |
|---|------|------|------|
| 1 | 功能规划前的竞品调研 | 新功能开始排期前，搞清楚竞品已经做了什么、做得好不好，避免做重复的东西。 | Mode A · 初始化 |
| 2 | 立项汇报需要竞品对比 | 老板要看横向对比，快速出一份有定位图、有优先级建议的竞品分析报告。 | Mode A · 初始化 |
| 3 | 竞品发布新版本，更新分析 | 竞品改版了，只需要更新对应部分，不重新做整份报告，版本号自动递增。 | Mode B · 增量更新 |
| 4 | 用户流失分析：为什么不选我们 | 从竞品口碑和差异化角度，找到用户选择竞品的真实原因，而不只是猜。 | Mode A · 初始化 |
| 5 | 市场进入前的行业全景调研 | 还没有自己的产品，只想了解行业现状和玩家格局。切换市场全景视角，取消己方列。 | 全景视角模式 |
| 6 | 持续跟踪竞品动态 | 建好基础文档后，每次有新竞品或竞品更新，一句话触发增量更新，文档持续保鲜。 | Mode B · 增量更新 |

### 支持的输入类型

| 输入类型 | 说明 | 处理方式 |
|---------|------|---------|
| 竞品名称 / 链接 | 直接告诉 AI 竞品叫什么，或提供官网/产品页链接，自动联网采集信息。 | WebSearch · WebFetch |
| 截图 / 截屏 | 上传竞品界面截图，AI 识别功能模块、UI 布局和关键信息，无需文字描述。 | 图像识别解析 |
| 文字体验笔记 | 把你自己体验竞品后写的笔记、会议记录或用户反馈直接粘贴，AI 提炼结构化信息。 | 直接解析（可信度最高） |
| 上传文档 | 上传已有的竞品研究报告、Excel 表格或 PDF，AI 读取并提取竞品信息继续分析。 | 文件读取解析 |

**可信度优先级：** 你自己的一手体验笔记 > 官网/产品页面 > 搜索结果/媒体报道

### 输出内容一览

**分析文件（Markdown，AI 处理源文件）**

| 文件 | 内容 |
|------|------|
| `competitive-analysis.md` | 完整竞品分析报告：功能对比矩阵、用户口碑对比、定位图、价值曲线+ERRC、机会点与行动建议 |
| `feature-matrix.md` | 功能矩阵独立文件，Mode B 增量更新时可精准更新单个竞品的列 |

**交付文档（可直接分发给决策者）**

| 文件 | 内容 |
|------|------|
| `竞品分析报告_v1.0.docx` | 宋体/黑体字体规范，含封面、变更记录、所有分析章节；旧版保留不覆盖 |
| `功能对比矩阵_v1.0.xlsx` | Sheet 1 功能矩阵 + Sheet 2 竞品概况，列宽自适应 |

### 方法论支撑

五层分析形成完整决策依据：

| 层次 | 方法 | 回答的问题 | 核心洞察 |
|------|------|-----------|---------|
| Layer 01 | 功能矩阵 | 竞品有没有这个功能 | 口碑差的从 ✅ 自动降为 🔶 |
| Layer 02 | 用户口碑 | 做得好不好 | 揭示官网宣传和用户实感的落差 |
| Layer 03 | 定位图 | 你在哪里 | 一眼看出市场扎堆区和空白区 |
| Layer 04 | 价值曲线 | 在哪里投入 | 蓝海战略框架，找红海和机会 |
| Layer 05 | ERRC 行动框架 | 该减什么 | 消除/减少/提升/创造，找真正差异化路径 |

### 与其他 PM Skill 的联动

```
pm-requirements → feature-list.md → pm-compete → 完整竞品分析报告
```

pm-compete 启动时自动检测 `_pm_context/requirements/feature-list.md` 是否存在。存在则直接读取己方产品信息，跳过"你们产品是什么"的追问，直接进入竞品信息采集环节。

### 典型触发方式

```
帮我做个竞品分析，把客如云、二维火和美团餐饮管家都分析一下          → Mode A
竞品出新版本了，二维火刚发布了新的跨店排班功能，帮我更新一下竞品文档  → Mode B
为什么用户不选我们？从竞品角度分析一下，最近流失率有点高             → Mode A
我想了解一下这个市场的情况，还没有自己的产品，先研究研究连锁餐饮管理 SaaS 的玩家格局 → 全景视角
帮我做个对标分析，要给老板汇报，需要功能矩阵和差异化建议             → Mode A
```

## Part 2 · 课程录制指引

### 录制前准备清单

**演示文件**
- [ ] 准备好 `feature-list.md`（推荐：使用 pm-requirements 生成的那份）
  路径：`_pm_context/requirements/feature-list.md`（相对于 Claude Code 工作目录）
- [ ] 选定 3 个演示竞品（推荐：客如云 / 二维火 / 美团餐饮管家）
- [ ] 可选：准备一张竞品截图，用于演示"截图输入"能力

**环境配置**
- [ ] 确认 Claude Code 已安装，pm-compete Skill 已就位（`~/.claude/skills/pm-compete/`）
- [ ] 确认网络正常，AI 可以联网搜索（WebSearch 功能需要）
- [ ] 新建一个干净的演示项目目录（如 `D:\demo-pm-compete\`）

**录制工具**
- [ ] 录屏工具就绪（OBS / Loom / Win+G），建议 1080P 以上，终端字体 ≥ 16px
- [ ] 关掉不相关通知和应用，保持桌面整洁

**演示策略**
- [ ] 先演练一遍，确认 AI 输出方向符合预期再正式录
- [ ] 推荐演示联动场景（有 feature-list.md），更能体现 Skill 系列价值

### 推荐录制流程（完整演示预计 10–15 分钟）

**Step 1 · 开场：设置场景（约 1 分钟）**
- 描述 PM 做竞品分析的传统痛点：手动搜、手动汇总、没方法论、结论说不清楚
- 展示演示目录结构，指出 feature-list.md 已存在（上一个 Skill 生成的）

> **提示：** 强调"这是 pm-compete 和 pm-requirements 的联动"，让学员理解 Skill 系列是一个完整工作流。

**Step 2 · 触发 Skill：自然语言输入（约 1 分钟）**
- 在 Claude Code 中输入触发语句
- AI 提示「已读取到你们的功能清单」—— 停一下解释这是自动读取 feature-list.md 的结果

> **高光提示：** 着重说"它没有问我们产品是什么，因为它已经读到了我们之前整理好的功能清单"

**Step 3 · 信息采集过程（约 2–3 分钟）**
- 回复 AI，提供三个竞品：客如云、二维火、美团餐饮管家
- 展示 AI 逐个执行 WebSearch，实时搜索官网、App Store 评价、知乎讨论
- 旁白：搜功能 + 搜用户评价 + 对比分析，传统做法要做三四个小时

**Step 4 · 分析结果：逐模块展示（约 5–6 分钟）**
- 打开 `competitive-analysis.md`，展示功能矩阵符号系统和备注列
- 用户口碑部分——着重说"不只是列功能，还告诉你功能做得好不好"
- 定位图——强调"一眼看清市场格局，知道我们在哪个象限"
- 价值曲线和 ERRC——强调"不只是加功能，也告诉你该减什么"
- 机会点建议——P0/P1/P2 优先级，每条结论直接可操作

> **节奏提示：** 不需要每个章节都讲透，每个模块点一个最有代表性的发现，说清楚价值，继续走。

**Step 5 · 确认并生成交付文件（约 1–2 分钟）**
- 确认内容，AI 生成 Word 报告和 Excel 功能矩阵
- 展示 .docx 格式规范（宋体/黑体、封面、变更记录）
- 收尾：一条 prompt，30 分钟内完成的竞品分析，换人工要做两三天

**Step + · 可选：演示 Mode B 增量更新（约 1–2 分钟）**
- 输入"假设竞品发了新功能"的增量更新触发语句
- 展示 AI 只修改对应部分，版本号从 v1.0 变成 v1.1

### 核心功能演示要点与参考话术

**1. 功能矩阵 — 「有没有」一眼看清**

> 「注意这里，同样是"外卖对接"，竞品都是✅，但我们系统在备注里指出——客如云的口碑显示实际同步经常出问题，所以我们标的是🔶，不是✅。这就是口碑和功能矩阵交叉验证的价值。」

**2. 用户口碑 — 揭示「官网 vs 真实」的落差**

> 「客如云官网写的是'智能管理'，但用户反复说的是'太复杂学不会'——这就是官方宣传和真实口碑之间的落差。我们只需要在易用性上做好，就能赢得从客如云流失的用户。」

**3. 定位图 — 一眼看出自己在哪**

> 「这个图的价值不是告诉你竞品有哪些，是告诉你我们在哪里。你看，我们和客如云根本不在同一个象限，我们服务的用户规模不同，这就不是正面竞争，是两个不同的市场。」

**4. ERRC 框架 — 不只是"加"，也要"减"**

> 「大家做竞品分析容易只想着"竞品有什么我们要跟"。但 ERRC 框架逼你同时想：哪些东西是竞品普遍有但其实用户不需要的？消除这些，才能真正做到简单。」

### 常见问题预案

| 问题 | 处理方式 |
|------|----------|
| AI 搜不到竞品详细信息，输出很多 ❓ | 演示亮点：「它在信息不足的格子里标了❓而不是乱填——不编造内容」。可把手动体验笔记粘给它补全。 |
| Skill 没有自动触发 | 检查触发词是否含"竞品分析"、"分析竞品"等关键词。可直接输入 `/pm-compete` 强制指定。 |
| AI 没有自动读取 feature-list.md | 检查路径：`_pm_context/requirements/feature-list.md`（相对于当前工作目录）。 |
| Word / Excel 生成失败 | Skill 有 fallback 机制，docx/xlsx 不可用时尝试 Python 脚本，Python 也不行则直接使用 .md 文件。 |
| AI 输出太长，录制节奏被打断 | 剪辑时做时间加速（x2-x4），或旁白「AI 正在生成完整报告，这里我们快进一下」。 |

### 课程脚本提示

**开场引导：**
> 「今天这个 Skill 叫 pm-compete，专门用来做竞品分析。传统做法是：搜竞品官网、手工汇总功能、整理成 Excel、再写分析结论——这个流程快则半天，慢则两三天。我们来看看 AI 怎么做。」

**联动亮点解说：**
> 「注意它没有问我们产品是什么——因为我们在上一节课用 pm-requirements 整理需求的时候，已经生成了一份功能清单。pm-compete 自动读取了那份文件，直接进入竞品分析环节。这就是 Skill 系列联动的价值：每个环节的产出，都是下一个环节的输入。」

**方法论价值解说：**
> 「这个 Skill 有五层分析：功能矩阵告诉你有没有，用户口碑告诉你做得好不好，定位图告诉你你在哪里，价值曲线告诉你在哪里投入，ERRC 框架告诉你该减什么。这五层加起来，才是一份真正能指导决策的竞品分析。」

**收尾总结：**
> 「我们刚才用了不到 30 分钟，完成了一份包含功能矩阵、用户口碑、定位图、价值曲线和 ERRC 行动框架的竞品分析报告，以及可以直接发给老板的 Word 文档和 Excel 表格。这份东西如果人工做，两三天起步。」
