# 敏捷项目管理框架参考

> 本文档供项目管理 Skill 在分析过程中参考选用，不作为报告章节直接输出。

## 一、Scrum 框架

### 核心角色
- **Product Owner（产品负责人）**：管理产品待办列表，决定优先级，对产品价值负责
- **Scrum Master**：保障 Scrum 流程执行，消除障碍，团队教练
- **Development Team（开发团队）**：自组织，跨职能，每个 Sprint 交付可用的增量

### 核心仪式
| 仪式 | 频率 | 时长建议 | 核心产出 |
|------|------|----------|----------|
| Sprint 规划会 | 每 Sprint 开始 | 2-4 小时 | Sprint 目标 + Sprint 待办列表 |
| 每日站会 | 每天 | 15 分钟 | 昨天做了什么、今天做什么、有没有阻碍 |
| Sprint 评审会 | 每 Sprint 结束 | 1-2 小时 | 可工作的产品增量 + 反馈 |
| Sprint 回顾会 | 每 Sprint 结束 | 1-1.5 小时 | 改进项（团队流程、工具、协作） |

### Sprint 周期建议
- **2 周**：最常用的节奏，平衡计划性和灵活性
- **1 周**：适合需求变化极快的探索型项目
- **3-4 周**：适合跨团队依赖多、发布风险高的项目

### 产品经理在 Scrum 中的价值
- 在 Sprint 规划前完成需求澄清，写清楚验收标准
- 在评审会上收集真实用户反馈，验证需求假设
- 在回顾会上了解团队痛点，协调资源解决
- 维护健康的 Product Backlog（粒度合适、优先级清晰、依赖标注）

## 二、看板方法

### 核心原则
1. **可视化工作流**：把所有任务放到看板上，让流动透明
2. **限制在制品数量（WIP Limit）**：防止团队同时做太多事，减少上下文切换
3. **管理流动**：关注任务的流动效率，而非资源利用率
4. **明确流程规则**：每个阶段的进入和退出条件要清晰
5. **建立反馈循环**：定期检视和调整

### 常用看板配置
| 列名 | 说明 | WIP 限制建议 |
|------|------|-------------|
| 待办（Backlog） | 已确认但未开始的任务 | 不限 |
| 进行中（Doing） | 正在开发 | ≤ 团队人数 × 0.8 |
| 测试中（Testing） | 开发完成等待验证 | ≤ 2 |
| 待发布（Ready） | 测试通过等待上线 | ≤ 3 |
| 已完成（Done） | 已上线 | 不限 |

### 关键度量
- **流动时间（Lead Time）**：从需求提出到交付的完整时间
- **周期时间（Cycle Time）**：从开始处理到完成的时间
- **吞吐量（Throughput）**：单位时间完成的任务数
- **累积流图（CFD）**：可视化各阶段任务数量的变化趋势，识别瓶颈

### 产品经理在看板中的价值
- 与团队共同定义 WIP 限制，防止过度承诺
- 定期检视累积流图，识别流程瓶颈
- 根据吞吐量合理预期交付时间，管理利益相关者预期
- 参与优先级排序，确保最重要的事情先被处理

## 三、混合模式

### 什么时候适合混用

| 场景 | 推荐模式 | 说明 |
|------|----------|------|
| 需求稳定、发布节奏固定 | Scrum | 有清晰的 Sprint 节奏 |
| 需求变化快、碎片化需求多 | 看板 | 不受迭代周期约束 |
| 多产品线/多团队协作 | Scrum of Scrums | 顶层用 Scrum，子团队可灵活选择 |
| 运维 + 开发并存的团队 | Scrum + 看板 | 开发走 Sprint，运维走看板 |
| 大型项目需要阶段性交付 | Scrum（长迭代） | 4 周迭代，每迭代交付一个大功能 |

### 混合实践建议
- **用 Scrum 的规划仪式**（规划会、评审会、回顾会）保证团队对齐
- **用看板的 WIP 限制**防止过载，特别是多项目并行时
- **用累积流图**替代燃尽图作为进度可视化工具（更真实）
- **用 Sprint 目标**代替任务完成率作为团队承诺

## 四、选择框架的决策依据

问自己三个问题：
1. **需求变化频率**：变化极快 → 看板；相对稳定 → Scrum
2. **团队规模**：小团队（≤7人）→ 看板或 Scrum 均可；大团队 → Scrum
3. **交付节奏要求**：固定节奏交付 → Scrum；持续交付 → 看板

没有银弹。选一个先用起来，在回顾会里调整。
