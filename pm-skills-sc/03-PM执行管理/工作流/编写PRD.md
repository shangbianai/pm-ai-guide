# /write-prd -- 产品需求文档 (PRD)

创建一个结构化的 PRD，用于对齐利益相关者并指导开发。接受从模糊的想法到详细的简报等任何形式的输入。

## 调用示例

```
/write-prd 为企业客户提供 SSO 支持
/write-prd 用户在入职流程的第 3 步大量流失 —— 我们需要修复它
/write-prd [上传简报、研究文档或战略 PPT]
```

## 工作流程

### 第 1 步：了解功能需求

接受任何形式的输入：
- 功能名称（"SSO 支持"）
- 问题陈述（"企业客户不断要求集中化身份验证"）
- 用户请求（"用户想要将数据导出为 CSV"）
- 模糊的想法（"我们应该针对入职流失做点什么"）
- 上传的文档（简报、研究报告、Slack 讨论记录、电子邮件）

### 第 2 步：收集背景信息

以对话方式询问 —— 首先提问最重要的，然后逐步填补空白：

1. **用户问题**: 这解决了什么问题？谁遇到了这个问题？痛点有多严重？
2. **目标用户**: 哪些用户细分群体？涉及多少人？他们目前的替代方案是什么？
3. **成功指标**: 我们如何知道这是否奏效？如果做成了，哪些指标会提升？
4. **约束条件**: 技术限制、时间表、监管要求、对其他团队的依赖？
5. **先前尝试**: 以前尝试过解决这个问题吗？市场上有现成的解决方案吗？
6. **范围偏好**: 是一步到位的完整方案，还是分阶段实施？

如果用户提供了包含背景信息的文档，请提取已有信息，仅针对缺失部分进行提问。

### 第 3 步：生成 PRD

应用 **create-prd** 技能，产出包含 8 个章节的文档：

```
## 产品需求文档: [功能名称]

**作者**: [用户]
**日期**: [今天]
**状态**: 草稿
**利益相关者**: [如果已知]

### 1. 执行摘要 (Executive Summary)
[2-3 句话：做什么、为谁做、为什么现在做]

### 2. 背景与上下文
[问题领域、前期研究、市场背景、是什么促使了该需求的产生]

### 3. 目标与成功指标
**目标** (成功是什么样子):
1. [具体的、可衡量的目标]
2. [...]

**非目标 (Non-Goals)** (明确排除在范围之外):
1. [我们不做的事情及其原因]
2. [...]

**成功指标**:
| 指标 | 当前值 | 目标值 | 衡量方式 |
|--------|---------|--------|-------------|

### 4. 目标用户与细分群体
[服务的对象、用户画像、细分市场规模]

### 5. 用户故事与需求

**P0 — 必须有 (Must Have)**:
| # | 用户故事 | 验收标准 |
|---|-----------|-------------------|

**P1 — 应该有 (Should Have)**:
| # | 用户故事 | 验收标准 |
|---|-----------|-------------------|

**P2 — 可以有 / 未来 (Nice to Have)**:
| # | 用户故事 | 验收标准 |
|---|-----------|-------------------|

### 6. 解决方案概述
[高层级方案、关键设计决策、技术方案（如果已知）]

### 7. 待解决问题
| 问题描述 | 负责人 | 截止日期 |
|----------|-------|----------|

### 8. 时间表与分阶段计划
[里程碑、依赖关系、分阶段实施计划（如果适用）]
```

### 第 4 步：评审与迭代

生成后，提供建议：
- "需要我 **收缩范围** 吗？我可以挑战哪些 P1 需求其实应该放到 P2。"
- "我是否应该针对该 PRD **进行事前剖析 (Pre-mortem)**？"
- "需要我 **将其拆解为用户故事** 提交给工程团队吗？"
- "我是否应该 **起草一份利益相关者更新** 来同步该计划？"

将 PRD 作为 markdown 文件保存到用户的工作区。

## 注意事项

- 对范围有明确的态度 —— 一份精炼的 PRD 优于一份庞大而模糊的 PRD
- 如果想法太大，主动建议分阶段实施，并仅针对第一阶段编写规格说明
- "非目标" 与 "目标" 同样重要 —— 它们可以防止范围蔓延 (Scope Creep)
- 成功指标必须具体："提升 NPS" 很糟糕，"在发布后 90 天内将 NPS 从 32 提升到 45" 很好
- 待解决问题应该是真正未决的 —— 不要列出那些可以从上下文推断出的内容
- 如果用户提供了研究资料，请将见解织入 "背景" 章节并注明出处
