# /pre-mortem -- 发布前风险分析

想象你的发布已经失败了。现在，倒推原因。此命令应用 "老虎/纸老虎/房间里的大象" (Tigers/Paper Tigers/Elephants) 框架来揭示真实风险并创建缓解计划。

## 调用示例

```
/pre-mortem [粘贴或上传 PRD、发布计划或功能规格书]
/pre-mortem 我们下个月要发布一个自助账单门户
```

## 工作流程

### 第 1 步：接收计划

接受任何格式：PRD、功能规格书、发布计划、项目简报或口头描述。提供的细节越丰富，风险分析就越精准。

### 第 2 步：风险识别

应用 **pre-mortem** 技能：

想象产品已经发布并失败了。生成以下类别的风险：
- **技术风险**: 性能、可扩展性、集成失败、数据问题
- **用户风险**: 采用障碍、易用性问题、未达预期
- **业务风险**: 收入影响、竞争对手反应、市场时机
- **运营风险**: 支持负载、文档缺失、培训需求
- **依赖风险**: 第三方服务、跨团队协作、合规监管

### 第 3 步：风险分类

对每个风险进行归类：

**老虎 (Tigers)** — 真实的、实质性的风险，可能导致失败
- 评估严重程度：阻碍发布 (Launch-blocking) / 快速跟进 (Fast-follow) / 持续跟踪 (Track)
- 对于 "阻碍发布" 的老虎：必须立即制定缓解措施
- 对于 "快速跟进" 的老虎：计划在发布后的第一个冲刺内解决
- 对于 "持续跟踪" 的老虎：进行监控，但不延迟发布

**纸老虎 (Paper Tigers)** — 听起来可怕但被夸大的风险
- 解释为什么该担忧是可控的
- 说明在什么情况下它会变成真正的 "老虎"

**大象 (Elephants)** — 团队心知肚明但避而不谈的潜在风险
- 揭示政治性、组织性或令人不适的风险
- 以建设性的方式提出建议的沟通切入点

### 第 4 步：生成事前剖析报告

```
## 事前剖析报告: [功能/发布项目]

**日期**: [今天]
**状态**: [草稿 / 已评审]

### 风险摘要
- **老虎 (Tigers)**: [数量] ( [阻碍发布], [快速跟进], [持续跟踪] )
- **纸老虎 (Paper Tigers)**: [数量]
- **大象 (Elephants)**: [数量]

### 阻碍发布的老虎 (Launch-Blocking Tigers)
| # | 风险描述 | 可能性 | 影响力 | 缓解措施 | 负责人 | 截止日期 |
|---|----------|--------|--------|----------|--------|----------|

### 快速跟进的老虎 (Fast-Follow Tigers)
| # | 风险描述 | 可能性 | 影响力 | 计划应对方案 | 负责人 |
|---|----------|--------|--------|--------------|--------|

### 持续跟踪的老虎 (Track Tigers)
[发布后需监控的风险及其触发条件]

### 纸老虎 (Paper Tigers)
[看起来严重但实际可控的担忧 —— 附带理由]

### 房间里的大象 (Elephants in the Room)
[团队应该讨论的那些令人不安的真相]

### 发布前检查清单 (Go/No-Go Checklist)
- [ ] 所有 "阻碍发布" 的老虎已得到缓解
- [ ] "快速跟进" 计划已记录并分配
- [ ] "持续跟踪" 的监控方案已到位
- [ ] 已定义回滚计划
- [ ] 已向支持团队进行简要说明
```

保存为 markdown 文件。

### 第 5 步：后续建议

- "需要我根据风险缓解措施 **更新 PRD** 吗？"
- "我是否应该为最危险的区域 **创建测试场景**？"
- "需要我根据这些发现 **起草一份发布检查清单** 吗？"

## 注意事项

- 进行事前剖析的最佳时机是计划完成 80% 时 —— 足够早以改变方向，足够晚以有实质内容
- 挖掘深层风险 —— 最危险的风险往往是没人提及的那些
- "大象" 是最有价值的输出 —— 揭示团队回避讨论的问题
- 对于每只 "老虎"，缓解措施应该是具体的、可分配的，而不仅仅是 "多加小心"
- 如果事前剖析揭示了过多的 "阻碍发布的老虎"，建议推迟或分阶段发布
