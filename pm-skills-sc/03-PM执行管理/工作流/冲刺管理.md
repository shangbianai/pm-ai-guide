# /sprint -- 冲刺生命周期

涵盖冲刺生命周期的三种模式：**plan** 用于冲刺计划，**retro** 用于回顾，**release-notes** 用于发布沟通。

## 调用示例

```
/sprint plan 2周冲刺，4名工程师，专注于结账流程优化
/sprint retro [粘贴团队反馈或冲刺数据]
/sprint release-notes [粘贴任务工单、变更日志或 PRD]
/sprint                    # 询问你当前处于哪个阶段
```

## 模式说明

---

### 计划模式 (Plan Mode)

通过容量估算、故事选择和风险识别为冲刺计划做准备。

#### 工作流程

**第 1 步：收集冲刺背景**
- 冲刺时长（1 或 2 周）
- 团队构成（工程师、设计师、QA —— 以及他们的可用性）
- 冲刺目标或重点领域
- 待考虑的待办事项 (Backlog Items)（粘贴、上传或描述）
- 上个冲刺的遗留任务
- 已知的干扰项（节假日、值班、会议）

**第 2 步：估算容量**

应用 **sprint-plan** 技能：

- 扣除会议、值班、请假后，计算可用的工程小时数/点数
- 根据历史数据（如果提供）或行业标准（理论容量的 70%）进行速率调整
- 显示每位团队成员的容量明细

**第 3 步：选择并排列故事顺序**

- 建议在容量范围内可以容纳哪些故事
- 标记依赖链（A 必须在 B 开始前完成）
- 识别风险：描述不详、有外部依赖或需要设计输入的故事
- 平衡 "快速见效" (Quick Wins) 与大型项目
- 确保每个故事都有验收标准

**第 4 步：生成冲刺计划**

```
## 冲刺计划: [冲刺名称/编号]

**时长**: [日期范围]
**冲刺目标**: [一句话概括]
**团队**: [成员及可用性]

### 容量
| 成员 | 可用天数 | 点数/小时 | 备注 |
|--------|--------------|-------------|-------|

**总容量**: [X] 点数/小时
**建议承诺**: [Y] 点数/小时 (预留缓冲)

### 选定故事
| # | 故事标题 | 点数 | 负责人 | 依赖项 | 风险 |
|---|-------|--------|-------|-------------|------|

### 冲刺风险
1. [风险] — 缓解措施: [行动]

### 完成定义 (Definition of Done)
- [ ] 代码已评审
- [ ] 测试已通过
- [ ] 已部署到预发环境 (Staging)
- [ ] QA 已批准
- [ ] 文档已更新（如果适用）
```

---

### 回顾模式 (Retro Mode)

协助进行结构化回顾，并产出可落地的改进方案。

#### 工作流程

**第 1 步：收集冲刺反馈**

接受以下形式的输入：
- 团队反馈（粘贴自调查问卷、Slack 或协作文档）
- 冲刺指标（速率、Bug 数量、事故记录）
- 用户自己的观察

询问："你更喜欢哪种回顾格式？"
- **开始/停止/继续 (Start/Stop/Continue)** (简单、快速)
- **4Ls** (Liked 喜欢, Learned 学到, Lacked 缺失, Longed for 渴望)
- **帆船 (Sailboat)** (风 = 助力, 锚 = 阻力, 岩石 = 风险, 岛屿 = 目标)

**第 2 步：分析与结构化**

应用 **retro** 技能：

- 将反馈归类到所选框架中
- 识别主题和模式
- 区分现象与根本原因
- 强调值得庆祝的胜利

**第 3 步：生成回顾总结**

```
## 冲刺回顾: [冲刺名称]

**日期**: [今天]
**格式**: [Start/Stop/Continue | 4Ls | Sailboat]
**参会者**: [如果已知]

### 做得好的地方
[按主题分组并附带证据]

### 做得不好的地方
[按主题分组并附带根本原因分析]

### 关键洞察
[涌现出的 2-3 个模式]

### 行动项
| # | 任务 | 负责人 | 截止日期 | 优先级 |
|---|--------|-------|----------|----------|

### 本冲刺指标
| 指标 | 本冲刺 | 上个冲刺 | 趋势 |
|--------|-----------|------------|-------|
```

---

### 发布说明模式 (Release Notes Mode)

将技术产出转化为面向用户的发布说明。

#### 工作流程

**第 1 步：接收发布内容**

接受：
- Jira/Linear 任务单或变更日志
- PRD 或功能规格书
- Git 提交信息或 PR 描述
- 团队内部对已交付内容的总结

**第 2 步：转化**

应用 **release-notes** 技能：

- 将技术术语转化为用户获益
- 分类为：新功能、改进、错误修复
- 以产品的语调进行编写（如果语调不明确请询问）
- 首先重点介绍影响力最大的变化

**第 3 步：生成发布说明**

```
## 更新日志 — [版本号/日期]

### 亮点
[用 1-2 句话总结最重要的变化]

### 新功能
- **[功能名称]** — [用通俗语言描述用户获益]

### 改进
- **[改进项]** — [哪些地方变得更好了]

### 错误修复
- 修复了导致 [用户影响] 的 [问题描述]

### 即将到来
[可选：下个版本的预告]
```

保存为 markdown 文件，并提供针对不同渠道（博客文章、应用内、电子邮件、Slack 公告）的格式化建议。

## 注意事项

- 对于计划模式：预留 20% 的缓冲时间处理计划外工作 —— 计划容量达 100% 的团队总是会失约
- 对于回顾模式：专注于 2-3 个高影响力的行动项，而不是列出 10 个没人会去做的任务
- 对于发布说明：始终将变化表述为用户获益，而非技术实现
- 各模式可以链式衔接：计划 → (冲刺执行) → 回顾 → 发布说明
