## 冲刺计划 (Sprint Planning)

通过估算团队容量、选择并排序故事以及识别风险来规划冲刺。

### 背景

您正在帮助 **$ARGUMENTS** 规划一次冲刺。

如果用户提供了文件（待办事项列表、速率数据、团队名单或之前的冲刺报告），请先阅读它们。

### 说明

1. **估算团队容量**：
   - 团队成员人数及其可用性（带薪休假 PTO、会议、值班）
   - 历史速率（过去 3 个冲刺中每个冲刺的平均故事点）
   - 容量缓冲：预留 15-20% 用于意外工作、错误和技术债
   - 以故事点或理想工时计算可用容量

2. **审查并选择故事**：
   - 从排好优先级的待办事项列表中拉取（最高优先级优先）
   - 验证每个故事是否符合“准备就绪的定义 (Definition of Ready)”（清晰的验收标准 AC、已估算、无阻碍）
   - 标记在承诺前需要进一步完善的故事
   - 当达到容量上限时停止添加故事

3. **映射依赖关系**：
   - 识别依赖于其他故事或外部团队的故事
   - 适当安排有依赖关系的故事顺序
   - 标记外部依赖关系及负责人
   - 识别关键路径

4. **识别风险与缓解措施**：
   - 具有高度不确定性或复杂性的故事
   - 可能延误的外部依赖
   - 知识过于集中（只有一个人能做）
   - 为每个风险建议缓解措施

5. **创建冲刺计划总结**：

   ```
   冲刺目标：[描述成功样貌的一句话]
   持续时间：[2 周 / 1 周 / 等]
   团队容量：[X 故事点]
   承诺的故事：[跨 Z 个故事，共 Y 个故事点]
   缓冲：[剩余容量]

   故事：
   1. [故事标题] — [点数] — [负责人] — [依赖关系]
   ...

   风险：
   - [风险] → [缓解措施]
   ```

6. **定义冲刺目标**：一个清晰、单一的句子，捕捉冲刺的主要价值交付。

逐步思考。保存为 markdown 格式。
