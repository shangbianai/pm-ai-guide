## 前端开发

**关注域：** 交互实现可行性、组件复用、状态管理、前端性能、浏览器兼容

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 组件拆分 | "这个页面的[X]和[Y]是否应该拆成独立组件？复用场景有哪些？" |
| 状态流转 | "加载中、空数据、错误、成功这四种状态是否都有设计？[X]场景下的过渡态怎么处理？" |
| 数据交互 | "前端需要调几个接口？是否支持并行请求？接口数据结构是否确定？" |
| 边界情况 | "[X]操作如果用户在过程中关闭页面或网络中断，数据会丢吗？" |
| 性能 | "列表数据量预估多大？是否需要虚拟滚动或分页？首屏加载怎么优化？" |
| 表单交互 | "表单校验规则是什么？实时校验还是提交时校验？校验失败后的提示方式？" |
| 权限控制 | "不同角色的页面可见性和操作权限是否有明确矩阵？前端如何判断？" |

### 严重度判断

- 🔴 **阻塞**：接口数据结构未定义、权限模型缺失、核心流程状态不全
- 🟡 **重要**：组件复用性未考虑、边界情况处理不完整、性能未评估
- 🟢 **建议**：交互细节可优化、加载动画可增强

## 后端开发

**关注域：** 数据模型设计、API 设计、系统性能、安全性、数据一致性

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 数据模型 | "这个功能涉及哪些数据实体？实体间关系是什么？是否有已有的数据表需要扩展？" |
| API 设计 | "接口是 RESTful 还是 GraphQL？需要支持批量操作吗？分页、排序、筛选的传参规范？" |
| 并发处理 | "[X]场景下有并发操作的可能吗？并发时如何保证数据一致性？" |
| 性能 | "预估的 QPS 是多少？是否需要缓存策略？数据库查询是否会成为瓶颈？" |
| 安全 | "是否有越权风险？敏感数据是否需要脱敏？操作是否需要审计日志？" |
| 数据迁移 | "新功能是否需要历史数据迁移？迁移策略是什么？回滚方案？" |
| 消息通知 | "是否需要异步处理或消息队列？通知的触发时机和频率？" |

### 严重度判断

- 🔴 **阻塞**：数据一致性无法保证、安全漏洞、API 设计不明确
- 🟡 **重要**：性能未评估、缓存策略缺失、数据迁移方案待定
- 🟢 **建议**：日志规范、监控埋点、接口文档格式

## 设计师

**关注域：** 视觉一致性、交互规范、可用性、信息架构、无障碍

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 设计一致性 | "这个页面/组件是否遵循现有设计系统？有没有新增的 UI 模式需要定义？" |
| 交互反馈 | "每个可操作元素是否有明确的 hover/active/disabled 状态？操作后的反馈方式是什么（toast/弹窗/页面跳转）？" |
| 信息层级 | "页面信息架构是否清晰？用户最关心的信息是否在第一屏？次要信息如何折叠？" |
| 极端情况 | "字段内容极长时怎么展示？列表为空时给用户什么引导？同时多个错误怎么呈现？" |
| 移动端适配 | "移动端是否有设计？响应式策略是什么（自适应/独立移动端）？手势操作是否需要考虑？" |
| 可用性 | "核心操作需要几步完成？是否有不必要的步骤可以合并？新用户是否需要引导？" |

### 严重度判断

- 🔴 **阻塞**：核心流程缺少关键状态设计、信息架构混乱
- 🟡 **重要**：交互反馈不完整、移动端未考虑、极端情况缺失
- 🟢 **建议**：微交互优化、动画细节、视觉层次微调

## 测试/QA

**关注域：** 测试场景覆盖、边界条件、异常路径、兼容性、性能测试

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 功能测试 | "核心功能的测试场景有哪些？正向流程和异常流程是否都有覆盖？" |
| 边界测试 | "输入边界值是什么？最大/最小/空值/特殊字符的处理方式？" |
| 兼容性 | "需要支持哪些浏览器和设备？最低版本要求是什么？" |
| 回归影响 | "这个功能对现有功能有什么影响？回归测试范围是什么？" |
| 数据测试 | "测试数据怎么准备？是否需要数据构造脚本？生产数据脱敏方案？" |
| 性能测试 | "并发用户数预估多少？是否需要压测？性能基准是什么？" |
| 安全测试 | "是否有 SQL 注入/XSS/CSRF 风险？权限绕过测试是否覆盖？" |

### 严重度判断

- 🔴 **阻塞**：核心流程无测试场景、安全测试缺失、回归范围不明确
- 🟡 **重要**：边界测试不完整、兼容性未评估、测试数据未准备
- 🟢 **建议**：自动化测试覆盖、测试用例管理方式

## 业务方/运营

**关注域：** 业务逻辑正确性、用户价值、运营效率、上线策略

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 业务逻辑 | "这个逻辑是否和实际业务流程一致？有没有遗漏的业务分支？" |
| 用户价值 | "目标用户真的需要这个功能吗？有没有用户反馈或数据支撑？" |
| 上线策略 | "上线后如何通知用户？是否需要灰度发布？回滚方案是什么？" |
| 运营支持 | "运营后台是否需要配套功能？数据报表需求是什么？客服是否需要培训？" |
| 效果度量 | "上线后怎么判断功能是否成功？核心指标是什么？数据埋点是否完整？" |
| 用户迁移 | "老用户如何过渡到新功能？是否需要数据迁移或使用引导？" |

### 严重度判断

- 🔴 **阻塞**：业务逻辑与实际流程不符、无效果度量方式
- 🟡 **重要**：上线策略不明确、运营配套缺失、用户迁移未考虑
- 🟢 **建议**：运营活动配合、客服话术准备、FAQ 更新

## 老板/管理者

**关注域：** 战略对齐、资源投入、时间规划、风险控制、竞争格局

### 提问模式

| 方向 | 典型问题模板 |
|------|-------------|
| 战略价值 | "这个功能和公司/产品今年的战略目标有什么关系？为什么优先级排在其他需求前面？" |
| 资源评估 | "预计投入多少人天？前端/后端/设计/测试各需要多少？有没有技术风险？" |
| 时间规划 | "什么时候能上线？如果 delay 了影响什么？有没有必须在某个节点前上线的硬约束？" |
| 风险控制 | "最大的风险是什么？如果失败了影响范围多大？有没有止损方案？" |
| 竞争分析 | "竞品有没有类似功能？我们是跟进还是差异化？" |
| 替代方案 | "有没有更简单的方案可以达到同样目的？不做这个功能有没有替代方式？" |

### 严重度判断

- 🔴 **阻塞**：与战略目标不一致、资源远超可投入范围、无明确时间节点
- 🟡 **重要**：风险未评估、竞争分析缺失、替代方案未调研
- 🟢 **建议**：ROI 测算细化、里程碑节点优化

## 问题数量下限

| 输入复杂度 | 每角色最少问题数 |
|------------|------------------|
| 简单（1-3 个功能点） | 3 个 |
| 中等（4-8 个功能点） | 4 个 |
| 复杂（9+ 个功能点或含原型） | 5 个 |
